这个仓库旨在用一个 Qwen2.5-VL-7B-Instruct 的基座模型进行训练，学习医学知识，训练推理能力。

## 🏗️ 阶段一：中文医学知识增量预训练 (Continued Pre-training)

### 1. 预训练动机 (Motivation)
通用大模型在面对垂直领域（如医学）时，常因缺乏专业语料覆盖而产生“幻觉”或术语理解偏差。本阶段通过 **增量预训练 (PT)** 任务，在 **Qwen2.5-VL-7B-Instruct** 的基础上注入海量中文医学核心教材，旨在：
* **语义对齐**：调整模型对医学专有名词（如病理机制、药物代谢）的概率分布。
* **逻辑重构**：使模型掌握医学文本特有的论述逻辑与因果关系。

### 2. 语料库构建 (Corpus Construction)
数据源自 `data_clean/textbooks/zh_paragraph/` 路径下的结构化纯文本教材。我们对 15 门以上的核心学科进行了清洗，保留了高质量的学术描述段落。

**涵盖学科清单：**
* **基础医学**：病理学、病理生理学、人体寄生虫学、局部解剖学。
* **临床医学**：内科学、妇产科学、儿科学、神经病学、精神病学、传染病学。
* **专科医学**：耳鼻咽喉头颈外科学、临床药理学、法医学。

### 3. 训练配置 (Training Configuration)
本阶段采用 **LoRA (Low-Rank Adaptation)** 技术进行轻量化参数更新，确保在注入知识的同时不破坏基座模型的通用对话能力。

#### 深度逻辑审计 (Critical Notes)

1. **极低 Loss 的双刃剑效应**：
   `0.0242` 的收敛 Loss 证明了基座模型（v1_base）对 SFT 指令集的极大宽容度。
   * **正面逻辑**：模型现在会非常“听话”，预测格式（如单选/多选）将极其标准。
   * **负面逻辑风险**：如此低的 Loss 可能意味着模型产生了**“指令僵化”**。如果 SFT 语料多样性不足，模型可能会丧失一部分 PT 阶段习得的灵活推理能力，变成一个只会按固定模板吐字的机器。

2. **吞吐量效率审计**：
   `9.69 samples/s` 证明在开启 Flash Attention 2 且 `cutoff_len` 为 2048 的情况下，你对 4090 的并行压力利用得非常好。相比 PT 阶段，SFT 阶段的计算密度更集中于 Token 预测而非权重漂移。

3. **后续实验对齐建议**：
   既然 SFT1 的 Loss 已经压得这么死，建议在后续的 **CMB 测试** 中，重点观测模型是否出现了“只会给答案，不会给解释”的情况。如果出现这种情况，说明 SFT 强度过大，后续实验需考虑降低 Epoch 数或将学习率调至 `5e-6`。

#### 3.3.2 sft2(rank16)

**实验综述**：本阶段进入了“大规模精细化对齐”阶段。实验核心逻辑在于利用极低的学习率（5e-6）与限制性较强的低秩空间（Rank 16），在不破坏 `v2_final` 基座底层知识分布的前提下，对 26 万条 CMB 医疗指令进行广谱拟合。实验运行耗时约 5.7 小时（20633s），在吞吐量与拟合深度间取得了极佳的工程平衡。

| :--- | :--- | :--- |

| **数据规模** | 26w Samples / 1.0 Epoch | 样本量级提升了 20 倍以上，单轮迭代即实现了对医疗问答分布的全局覆盖。 |

1. **Loss 值的真实性审计**：
   `0.4107` 的 Loss 值是这份报告中最具含金量的指标。
   * 对于如此庞杂的 cmb_sft 数据集，如果 Loss 依然是 0.02，那说明模型已经完全崩溃。目前的 0.41 说明模型在尝试理解医学逻辑，且在验证集上具有极强的鲁棒性。

2. **吞吐量与数据切分**：
   `cutoff_len: 1024` 的设定在 26w 数据的规模下非常合理。它在物理上兼顾了 4090 的显存压力，使得 `per_device_train_batch_size: 4` 与 `accumulation: 8` 能够稳定运行。

#### 深度逻辑审计 (Critical Notes)

1. **“知识泵入”的副作用审计**：
   SFT3 采用了 Rank 128 这种“重型武器”来处理 2.5 万条填空数据。
   * **逻辑缺陷**：虽然 1.13 的 Loss 意味着模型记住了这些填空答案，但这种“纯召回”任务与医学考试的“排除法逻辑”并不兼容。
   * **后续验证**：根据之前的评测，此阶段导致模型变得“唠叨”（语义找回激增），证明高秩训练强行将模型的注意力从“选项字母”拉向了“名词解释”。

2. **工程代价评估**：
   为了塞进 40GB 显存，你牺牲了上下文长度（512）。
   * **风险判定**：这导致模型在 SFT3 期间丧失了对长病案（Case Report）的整体感知能力。如果医学题目超过 500 个 Token，SFT3 权重的处理能力将出现显著下降。

3. **模型身份的迷失**：
   `system_prompt` 的缺失或变更在这一阶段最为致命。由于 SFT3 的数据主要是“填空”，它冲淡了 SFT2 建立的“判别器身份”，直接导致了后续 CMB 测试中 Grand Total Accuracy 的下滑。

#### 🧪 4.1.1 SFT2：极致的指令稳健性
* **审计判定**：SFT2 表现出极强的指令韧性。无论使用默认助手提示词还是严厉的学术指令，**语义找回始终保持为 0**。这说明 SFT2 已经形成了牢固的单字母输出映射。

#### ⚠️ 4.1.2 SFT3：逻辑发散与“补丁”效应
* **审计判定**：SFT3 在默认状态下极度不稳定（语义找回 250）。P-Academic 虽然起到了一定的“禁言”作用（找回降至 75），并让正确率回升约 1.77%，但其本质仍是“带病运行”。
* **结论**：提示词工程无法挽救底层逻辑的损坏。SFT3 在面对复杂指令时产生的震荡，反映了填空式训练对参数分布的污染。

#### 4.2.2 深度逻辑审计 (Logic Audit)

 1. “潜力释放”与“格式束缚”的辩证逻辑
* **Base 模型 (82.81%)**：通过“深度逻辑模糊匹配”脚本，我们发现原模型拥有极强的医学推理潜力，但其 **90.13%** 的正确答案都深藏在长篇大论的末尾或思维链（CoT）中。
* **SFT1 模型 (84.24%)**：在准确率提升 1.43% 的同时，彻底解决了格式坍塌问题。其语义找回仅为 3 例，证明模型已形成“见题即吐字母”的条件反射。

2. 推理成本的降维打击
* **数据观察**：SFT1 的推理速度是 Base 模型的 **11.5 倍**。
* **物理本质**：Base 模型在每一题上都进行了大量的 Token 生成（$2339s \div 3426$ 题），导致显卡处于高负载低产出状态。SFT1 通过 SFT 训练成功引入了“逻辑截断”，在保持更高准确率的同时，极大地缩短了 KV Cache 的增长路径。
